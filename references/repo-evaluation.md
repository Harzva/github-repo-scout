# Repository Evaluation

Use this checklist when comparing GitHub candidates.

## Fit

- Does the repo directly address the user's task?
- Is it an implementation, a library, a template, a prompt/workflow, or only a concept?
- Is the technology stack compatible with the current project?

## Trust

- Recent pushes and releases are positive signals.
- Clear docs, examples, tests, and install instructions matter.
- A permissive license is important if code may be reused.
- Archived repos, missing licenses, broken demos, and vague setup steps are risks.

## Reuse

- Prefer repos that teach a pattern or provide a working reference.
- Save source cards for repos likely to help future tasks.
- Do not archive search noise just because a repo is popular.

## Output Shape

When reporting results, include repository name, URL, one-line reason it matters, risks, and a recommendation.
